    **Step 1 — Semantic HTML landmarks in index.html (criterion C7):** Wrap the existing DOM structure in semantic landmarks. The current structure uses bare <div> elements. Convert:
    - The viewport container `<div id="viewport">` becomes `<section id="viewport" role="application" aria-label="3D model viewport — drag rings to rotate, or use arrow keys" tabindex="0">`. The role="application" tells screen readers to pass keystrokes through (appropriate for a 3D canvas with custom keyboard handlers per research Pattern 6).
    - The left panel `<div class="panel panel-left" id="panel-left">` becomes `<aside id="panel-left" class="panel panel-left" aria-label="File and configuration">`.
    - The right panel `<div class="panel panel-right" id="results">` becomes `<aside id="results" class="panel panel-right" aria-label="Score and candidates">`.
    - Add a `<header>` wrapping the h1 and subtitle inside the left panel: `<header><h1>Orient STL</h1><p class="subtitle">...</p></header>`.
    - The main content area (viewport + status) should be wrapped in `<main>`. Since the viewport is position:fixed full-screen, the <main> wraps it semantically: `<main><section id="viewport" ...>...</section></main>`.
    - Add an `aria-live="polite"` region for orientation announcements: `<aside id="status-live" aria-live="polite" class="sr-only"></aside>` placed inside <main> after the viewport section. The .sr-only class was created in Plan 01's main.css.
    - Add an `id="viewport-canvas-label"` or use the aria-label directly on the section.

    **Step 2 — Keyboard rotation handler (criterion C6):** Add keyboard event handling to DragHandler (or a thin KeyboardHandler — but per research Pitfall 1, prefer folding into DragHandler to avoid micro-classes). Add an `attachKeyboardHandler(el: HTMLElement)` method to DragHandler that listens for 'keydown' on the viewport section element. The handler maps:
    - ArrowLeft → applyKeyRotation('axis-y', -step) (yaw ring, which maps to world Z per Pitfall 3)
    - ArrowRight → applyKeyRotation('axis-y', +step)
    - ArrowUp → applyKeyRotation('axis-x', -step) (pitch ring)
    - ArrowDown → applyKeyRotation('axis-x', +step)
    - 'q'/'Q' → applyKeyRotation('axis-z', -step) (roll ring, which maps to world Y per Pitfall 3)
    - 'e'/'E' → applyKeyRotation('axis-z', +step)
    - Step size: STEP_DEG (5 degrees) from constants.ts normally, FINE_DEG (1 degree) when Shift is held (e.shiftKey). Import both from constants.ts.
    - Any other key: return early (do not preventDefault).
    - On matched key: call e.preventDefault() to prevent page scroll.

    The applyKeyRotation method applies a quaternion rotation around the given axis (using the SAME axis mapping as getAxisVector — axis-y → world Z, axis-z → world Y per Pitfall 3) by the given angle, updates the mesh quaternion, calls colorOverhang, and invokes onOrientationChange with the new quaternion. This mirrors the pointer drag's rotation logic.

    Wire the keyboard handler in Viewport: in enterOverlayMode, call this.dragHandler.attachKeyboardHandler(this.renderer.domElement.parentElement). In exitOverlayMode or dispose, remove the keyboard listener. The viewport section element (which has tabindex=0) is the event target — it must have focus for keydown to fire.

    **Step 3 — aria-live announcements:** In AppController's updateLiveScore method (or wherever onOrientationChange is handled), update the #status-live element's textContent with the current orientation. Format: `Yaw ${yawDeg}°, Pitch ${pitchDeg}°, Roll ${rollDeg}°` or simply `Score: ${score}%, Profile: ${profileLabel}`. This announces changes to screen-reader users without stealing focus. The aria-live="polite" means the announcement waits for the user to pause, not interrupting.

    **Step 4 — Focus indicator CSS (criterion C6):** Add to web/src/styles/main.css: `#viewport:focus-visible { outline: 2px solid var(--color-accent); outline-offset: -2px; }`. This ensures keyboard users can see when the viewport has focus. Also add `#viewport { outline: none; }` as base to remove the default ugly outline, replaced by the focus-visible rule.

    **CRITICAL — Pitfall 3 axis consistency:** The keyboard handler MUST use the same getAxisVector mapping as the pointer drag. ArrowLeft/ArrowRight rotate the 'axis-y' ring (which maps to world Z). Do not independently re-derive the axis mapping — call DragHandler.getAxisVector('axis-y') to get the vector, then apply rotation around it. This guarantees keyboard and pointer produce the same rotation direction.
