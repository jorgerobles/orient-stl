# Context: Phase 3.5 — Scoring Expansion & Refinement

**Capturado:** 2026-07-13
**Origen:** PrusaSlicer codegraph comparison revealed missing heuristics + unweighted consensus concern + non-deterministic refine
**Status:** In progress

## Domain

This phase expands orient-stl's scoring and refinement to full parity with
PrusaSlicer's auto-orient and addresses architectural concerns surfaced
during the comparison.

### Sub-plan 01 — H5/H6 backfill (COMPLETE)

Two heuristics absent from orient-stl were discovered by comparing against
PrusaSlicer's `SLA/Rotfinder.cpp` (indexed via codegraph):

1. **H5 — Surface quality (axis-misalignment)**: PrusaSlicer's "Best surface
   quality" method maximises `area × (|n·X| + |n·Y| + |n·Z|)` — rewards faces
   not aligned to a single frame axis (discourages big flat shelves/walls).
2. **H6 — Print height**: PrusaSlicer's "Lowest Z height" method minimises the
   rotated bbox Z. Already computed but only used as a fudge factor; promoted to
   first-class metric.

Both rankers extended so all five heuristics participate in the composite.

### Sub-plan 02 — Profiles, seeded refine, TOPSIS (IN PROGRESS)

The comparison also surfaced three architectural issues that this sub-plan
addresses:

1. **Weight profiles externalised to JSON** — `WEIGHT_PRESETS` is hardcoded in
   `compute.ts`; externalising to `web/src/profiles/*.json` makes them
   user-editable without touching source.
2. **Seeded deterministic refine + variance metric (H7)** — `refine_orientation`
   uses `js_sys::Math::random()` (non-deterministic); replacing with a seeded
   xorshift32 PRNG makes results reproducible per click. Running K=4 seeded
   refinements per direction and computing the score variance gives a new
   "refine stability" metric (H7) — low variance means the direction is a
   stable basin, high variance means it's a noisy ridge.
3. **TOPSIS ranker + UI switcher** — `rankByConsensus` (minimax with equal
   weights) is brittle (dictatorial veto). TOPSIS is the textbook MCDA fix:
   it computes the geometric distance from the ideal-best and worst
   simultaneously, weighted per-metric. A UI dropdown lets users switch between
   rankers and weight profiles at runtime.

## Decisiones clave

| Decisión | Valor |
|----------|-------|
| H5 convention | MAXIMISE (higher = better, mirrors PrusaSlicer's `to_max()`); inverted to cost form inside rankers |
| H6 source | Reuse existing `computeHeight` (TS) / new `min_z_height` (Rust); PrusaSlicer uses convex hull bbox Z — equivalent since mesh vertex extremes == hull extremes |
| Consensus height handling | Replaced the `overhang × (1 + hN×0.5)` multiplicative fudge with height as an independent cost term in the max() — more principled |
| PRNG choice | xorshift32 — no crate dependency, ~5 ns/call, full 2³²-1 period, seedable from direction hash |
| K=4 batch refine | Balance: 4 runs is enough for a meaningful stddev without 4× latency; total refine still < 200ms for typical meshes |
| TOPSIS inclusion | Minimax (consensus) is brittle — dictatorial veto by any single metric; TOPSIS is the textbook MCDA fix with per-metric weights |
| Profile externalisation | JSON files under `web/src/profiles/` loaded via `import.meta.glob` — bundled at build time, no runtime fetch needed |
