  <files>core/src/candidates.rs, core/src/lib.rs</files>
  <action>
    **In `core/src/candidates.rs`:**

    Add two new public functions:

    1. `generate_fibonacci_sphere(n: usize) -> Vec<[f32; 3]>` — generates `n` evenly-distributed unit-direction samples on S² using the Fibonacci sphere algorithm. The golden ratio φ = (1.0 + √5) * 0.5. For i in 0..n: θ = acos(1 - 2*(i+0.5)/n), φ = 2π * i / φ, then compute [sin(θ)*cos(φ), sin(θ)*sin(φ), cos(θ)]. Use the `i` as seed for deterministic output.

    2. `generate_hull_plus_sphere(hull: &ConvexHull, n: usize, dedupe_angle_deg: f32) -> Vec<[f32; 3]>` — combines hull face normals with Fibonacci sphere samples (n points), then deduplicates with the existing `deduplicate_directions()` at the given angular threshold. Strategy: first deduplicate hull normals (as currently), then run Fibonacci, then deduplicate the combined set against the already-deduplicated hull set. This preserves all hull normals and adds only Fibonacci directions that are > dedupe_angle_deg away from any hull normal.

    The dedup strategy per the agent's discretion: generate N Fibonacci directions, then combine with hull normals and pass through `deduplicate_directions()` with the same 3° threshold used for hull-only mode. Fibonacci directions within 3° of a hull normal are dropped (they're redundant); those further away are added as new candidates. No special handling beyond the existing angular dedup.

    **In `core/src/lib.rs`:**

    Modify `prepare_data()` to handle `mode: "hull_plus_sphere"`:

    - When `config.mode == "hull"` (default): existing behavior unchanged.
    - When `config.mode == "hull_plus_sphere"`: call `generate_hull_plus_sphere(&hull, 200, config.dedupe_angle_deg)` instead of the two-step `generate_candidates + deduplicate_directions`.
    - Still reject unknown modes, but accept "hull_plus_sphere" in the check.

    The config deserialization already happens via `serde` with `#[serde(default = "default_mode")]`. The mode field is already present in `OrientConfig`. No changes needed to the struct.

    Do NOT change `OrientConfig`, the `OriData` output, or any existing JS-exposed interface. The only change to `prepare_data` is the mode branching.

    **Do NOT** add `wasm_bindgen` or make anything pub outside of `crate`. These functions stay `pub(crate)` because they're called from `lib.rs`.

  <name>Task 2: Hill-climb refine_orientation() WASM function + WASM build</name>
  <files>core/src/lib.rs</files>
  <action>
    **In `core/src/lib.rs`:**

    Add a new `#[wasm_bindgen]` function:

    ```rust
    #[wasm_bindgen]
    pub fn refine_orientation(
        positions: &[f32],
        normals: &[f32],
        areas: &[f32],
        dir_x: f32,
        dir_y: f32,
        dir_z: f32,
        critical_angle_deg: f32,
        iterations: u32,
    ) -> Vec<f32>
    ```

    Implementation:

    1. Build `MeshData` from the flat arrays using the existing `mesh::precompute_mesh` (which accepts `&[f32]` positions and returns a `MeshData` with normals/areas/vertices). **However**, since we already have precomputed normals and areas, use a lightweight helper that reconstructs `MeshData` without recomputing normals/areas. Actually — simplest path: the raw `prepare_data` output (positions, normals, areas) are per-triangle arrays. Use them directly by constructing a `MeshData` with the provided values. The mesh data structure expects vertices in a specific layout.

    Wait — `MeshData` stores `normal: Vec<[f32;3]>`, `areas: Vec<f32>`, `vertices: Vec<[f32;3]>`, `triangle_count: usize`. The flat arrays from JS are already in the right shape. Construct `MeshData` directly by chunking flat arrays:
    ```rust
    let tri_count = normals.len() / 3;
    let normals_vec: Vec<[f32; 3]> = normals.chunks_exact(3).map(|c| [c[0], c[1], c[2]]).collect();
    let areas_vec: Vec<f32> = areas.to_vec();
    // positions are per-triangle vertex groups (9 entries per triangle = 3 verts × 3 coords)
    let vertices_vec: Vec<[f32; 3]> = positions.chunks_exact(3).map(|c| [c[0], c[1], c[2]]).collect();
    let mesh = MeshData { normals: normals_vec, areas: areas_vec, vertices: vertices_vec, triangle_count: tri_count };
    ```

    2. Initialize current direction `dir = [dir_x, dir_y, dir_z]` normalized. Score it with `scoring::score_candidate(&dir, &mesh, critical_angle_deg)` to get baseline score.

    3. Hill-climb loop (iterations times):
       - Generate a random perturbation: small angular delta in a random orthogonal direction.
       - Implementation: use a fixed seed Rust RNG (e.g., `rand::rngs::StdRng::seed_from_u64(42)` — or use a simple deterministic hash-based perturbation to avoid adding the `rand` crate dependency). Per the agent's discretion: use `js_sys::Math::random()` via `wasm_bindgen` to get random values without adding the `rand` crate. Import via `use js_sys;` and call `js_sys::Math::random()`.
       - Perturbation: generate a random unit vector perpendicular to current dir, scale by a random angle in [-perturbation_deg, perturbation_deg] (start with 10° perturbation, reduce by `* 0.95` each iteration).
       - Rotate current dir by this perturbation on S²: `new_dir = normalize(current_dir + perturbation * angle)`, then renormalize.
       - Evaluate `new_score = scoring::score_candidate(&new_dir, &mesh, critical_angle_deg)`.
       - If `new_score < best_score` (lower = better, it's a penalty), accept: update `best_dir`, `best_score`.
       - Optionally decay perturbation angle each iteration (simulated annealing style) for convergence.

    4. Return `vec![best_dir[0], best_dir[1], best_dir[2], best_score]`.

    Note: `MeshData` and `score_candidate` are `pub(crate)` in their respective modules, so `lib.rs` can access them (both are in the same crate).

    The function SHOULD be exported to JS via `#[wasm_bindgen]`. No changes to `Cargo.toml` are needed (wasm-bindgen + js-sys already in dependencies).

    **Add to Cargo.toml** if `js-sys` is not already in the `[dependencies]` section. Check first — if `js-sys` is present, no change needed.
