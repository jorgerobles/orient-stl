## Interaction Contracts

### 1. Hull+Sphere Toggle

| Property | Spec |
|----------|------|
| Location | Inside `.config-panel`, existing config section. After critical angle slider. |
| Control type | Checkbox (`<input type="checkbox">`) — styled to match existing UI scale |
| Label | `Hull+Sphere Mode` with subtitle `Include ~200 Fibonacci-sphere candidates` |
| Default state | Unchecked (hull-only, backward compatible) |
| Interaction | On change: re-spawns computation with new mode (hull+sphere generates ~200 extra directions) |
| States | Checked / unchecked. Visual: uses `#4a90d9` accent when checked. |
| Re-compute trigger | On toggle: calls existing `spawnCompute()` with updated config. Shows progress bar. |

### 2. Overlay Drag-to-Rotate

| Property | Spec |
|----------|------|
| Entry | Click on a candidate `<li>` in the results list **OR** click on the 3D model itself |
| Visual entry | Viewport container gets a 2px `#4a90d9` border. Semi-transparent black backdrop (`rgba(0,0,0,0.65)`) fades over viewport area. Toolbar appears at top of viewport. |
| Backdrop | `position: absolute` overlay div covering the viewport container. Pointer events: `none` on backdrop so model receives drag events. Toolbar/buttons have pointer-events: auto. |
| Toolbar | Horizontal bar at top of overlay, 44px height, `rgba(0,0,0,0.8)` background, white text. Contains: "Overlay Mode" label, Esc hint, score badge, Varita Mágica button. |
| OrbitControls | **Disabled** while overlay is active (`controls.enabled = false`) |
| Drag interaction | User clicks and drags on the model surface. THREE.js `Quaternion` is updated incrementally from mouse delta (Arcball-style rotation around camera-relative axes). Mouse-down on model starts rotation; mouse-move updates quaternion; mouse-up stops. |
| Rotation pivot | Mesh centroid (existing baked geometry center — already correct from `viewport.ts`). |
| Update frequency | On every `mousemove` during drag: re-score from the closest pre-computed candidate direction OR approximate score from surrounding candidates. See note below. |
| Score badge position | Top-right of viewport overlay, fixed. A pill/badge: `background: rgba(0,0,0,0.75)`, `border-radius: 6px`, `padding: 4px 10px`. |
| Score badge content | `Score: {n}%` — white "Score:" label (12px), orange `{n}%` value (20.8px, 600 weight). |
| Exit | Press `Escape` key **OR** click "Exit" button in toolbar. On exit: backdrop fades out, border removed, OrbitControls re-enabled, model stays at last rotation. |
| Edge case — no drag | If user clicks but doesn't drag (mouseDown + mouseUp at same spot), treat as no-op — do NOT change rotation. |

### 3. Varita Mágica Button

| Property | Spec |
|----------|------|
| Label | `✨ Varita Mágica` |
| Location | Inside overlay toolbar, right-aligned. After score badge. |
| Style | `.button-success` base (`#27ae60`), white text, rounded 6px. Slightly larger padding: `0.5rem 1.5rem` for visual prominence. |
| Interaction | Click → calls `refineOrientation()` in Rust WASM via `wasm-bindgen`. Passes current quaternion + mesh data. WASM runs S² hill-climb (50 iterations default) and returns new quaternion + scores. |
| During compute | Score badge shows `Refining...` with subtle pulse animation. Button is disabled (`pointer-events: none`, opacity 0.6). |
| On complete | Score badge updates with new score. Mesh rotates to new quaternion. Overlay stays active so user can continue drag-refining. Button re-enables. |
| Error state | If WASM returns error: status bar shows "Hill-climb failed" for 3 seconds, badge reverts to previous score, button re-enables. |

### 4. Overlay Mode State Machine

```
[Idle] ──click candidate──→ [Overlay: Drag mode]
                               │
                               ├── mouseDown on model → drag starts
                               │   └── mouseMove → update quaternion + score badge
                               │   └── mouseUp → drag ends
                               │
                               ├── click "Varita Mágica" → [Computing hill-climb]
                               │   └── WASM returns → update quaternion + badge
                               │   └── error → revert, show error
                               │
                               ├── Esc / Exit button → [Idle]
                               │
                               └── new file drop → [Idle] (cancel overlay)
```

## States & Edge Cases

| State | Handling |
|-------|----------|
| No candidates loaded | Hull+sphere toggle and candidate click disabled (existing empty state) |
| Overlay active + file drop | Cancel overlay, show load progress, run new computation |
| Overlay active + nav buttons | Prev/Next buttons disabled during overlay mode (hidden or grayed) |
| Drag extends beyond viewport | `pointermove` listener on `document` (not just viewport) — rotation continues even if cursor leaves viewport |
| Hill-climb while dragging | Button disabled — user must finish drag first |
| Hull+sphere toggle while computing | Cancel current compute, start new compute with updated mode (existing pattern) |
| Rapid toggle clicks | Debounce by checking `cancelCompute` — each toggle cancels prior compute |

## Implementation Constraints

| Constraint | Detail |
|------------|--------|
| No new dependencies | All overlay/DOM elements created programmatically via `document.createElement` — same pattern as Phase 2 |
| Score during drag | Must interpolate or approximate from pre-computed candidates — DO NOT re-score full mesh per frame. Use: find nearest candidate direction via angular distance, show its score. |
| Overlay z-index | Overlay elements at `z-index: 10` relative to viewport container (which is `position: relative`) |
| CSS scoping | All new styles go into `index.html` `<style>` block (existing pattern), OR a new dedicated css file imported from `main.ts` |
| Maximum overlay elements | 1 toolbar + 1 backdrop + 1 score badge — no nested modals |
| Keyboard | Esc to exit overlay (documented). No other keyboard shortcuts in this phase. |
