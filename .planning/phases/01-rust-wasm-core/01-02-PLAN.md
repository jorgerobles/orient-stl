    Create `core/src/hull.rs`:
    - Vendored incremental quickhull algorithm ~300 lines, f32 precision
    - `pub(crate) struct ConvexHull { pub face_normals: Vec&lt;[f32; 3]&gt; }`
    - `pub(crate) fn compute_hull(vertices: &amp;[[f32; 3]]) -&gt; ConvexHull`
    - Implementation approach:
      1. Find extremal points (min/max x, y, z) → initial simplex (tetrahedron)
      2. For each vertex, assign to a visible face of the current hull (distance from face plane)
      3. For each face with assigned vertices, find the furthest vertex → creates new horizon edges
      4. Build new faces from horizon edges + furthest vertex
      5. Repeat until all vertices are inside the hull
      6. Extract face normals from the final hull (outward-facing, unit length)
    - Edge cases to handle (per orient-spec.md quickhull section):
      - Coplanar points: skip vertices that lie on the exact face plane (within epsilon 1e-6)
      - Degenerate hull: if all points are coplanar, hull is degenerate — return empty face_normals
      - Use f32 throughout — no f64 conversions needed
    - NOTE: The quickhull algorithm is well-documented. The implementation should follow the standard incremental approach:
      - Initial simplex from extreme points
      - For each remaining point, find which faces it is outside of
      - Compute horizon edges (edges of visible faces that border non-visible faces)
      - Build new faces from horizon + point
      - Remove old visible faces
    - IMPORTANT: Keep it minimal (~300 lines as spec'd). Comments explaining geometry steps are fine. Don't add defensive checks beyond epsilon handling.

    Create `core/src/candidates.rs`:
    - `pub(crate) fn generate_candidates(hull: &amp;ConvexHull) -&gt; Vec&lt;[f32; 3]&gt;`
      - Returns hull face normals as candidate directions (down directions in S² space)
      - Each normal is a unit vector pointing outward from the hull
      - Invert/negate to get the "down" direction? NO — the normal IS the direction to evaluate. See orient-spec.md scoring section: `down_local = R⁻¹ · (0, 0, -1)` and `cos_i = dot(n_i, down_local)`. The hull face normals represent directions to orient as "down." 
      - Actually re-reading: The hull face normals ARE candidate directions for the "down" vector. If a face normal points in direction D, orienting the model so D points downward makes that face a flat base. So we use hull normals directly as down_local candidates.
    - `pub(crate) fn deduplicate_directions(directions: &amp;[[f32; 3]], angle_deg: f32) -&gt; Vec&lt;[f32; 3]&gt;`
      - Merges direction vectors that are within `angle_deg` angular proximity
      - Algorithm: for each direction, compute dot product with all kept directions. If dot >= cos(angle_deg_rad), skip as duplicate
      - Use f64 for the angular comparison (acos of dot product) to avoid precision issues in polar clustering — per orient-spec.md warning in "Key pitfalls to avoid"
      - Preserves the first occurrence of each cluster
    - `pub(crate) fn compute_default_yaw(direction: &amp;[f32; 3], hull: &amp;ConvexHull) -&gt; [f32; 4]`
      - Computes the default quaternion for this candidate direction
      - Takes the candidate direction (down_local in world space) and computes a quaternion that rotates Z-up to align (0,0,-1) with this direction
      - For yaw: compute the bounding box of hull vertices projected onto the plane perpendicular to direction, find orientation that minimizes XY extent via rotating calipers
      - Return quaternion as [f32; 4] (w, x, y, z — standard wasm-bindgen quaternion format)
      - If rotating calipers is too complex for this task, use a simple heuristic: align one of the hull's longest edges in the perpendicular plane with the X-axis for yaw
      - IMPORTANT: ORIENT-08 requires this. The default yaw must minimize the XY bounding box for better platform utilization.

    Create `core/src/scoring.rs`:
    - `pub(crate) struct ScoreConfig { pub critical_angle_deg: f32 }`
    - `pub(crate) fn score_candidate(direction: &amp;[f32; 3], mesh: &amp;MeshData, critical_angle_deg: f32) -&gt; f32`
      - Implements the overhang penalty formula from orient-spec.md:
        ```
        cos_i = dot(n_i, direction)   // direction is "down_local" in S² space
        penalty_i = A_i * max(0, cos_i - cos(criticalAngle))
        score = Σ penalty_i
        ```
      - `direction` is the candidate down direction (S² unit vector)
      - `n_i` is the precomputed normal of triangle i
      - `A_i` is the precomputed area of triangle i
      - `critical_angle_deg` converted to radians: `theta = critical_angle_deg * PI / 180.0`
      - Uses f32 for area computation but f64 for angle trig (acos/cos) to avoid precision issues per orient-spec.md pitfall note
      - Only triangles with cos_i > cos(criticalAngle) contribute penalty — these are faces that point more downward than the critical angle threshold
      - Returns total area-weighted penalty across all triangles
    - Score is raw overhang penalty (lower is better)

    Create `core/src/stability.rs`:
    - `pub(crate) struct StabilityResult { pub stable: bool, pub margin: f32, pub contact_area: f32 }`
    - `pub(crate) fn check_stability(direction: &amp;[f32; 3], mesh: &amp;MeshData, hull: &amp;ConvexHull) -&gt; StabilityResult`
      - Algorithm from orient-spec.md §3:
        1. Find z_min: project all vertices onto the direction axis, find the minimum
        2. Find contact footprint: project all vertices at z_min ± epsilon onto the XY plane (perpendicular to direction)
        3. Compute 2D convex hull of footprint points
        4. Compute center of mass (CoM) position projected onto XY: approximate using centroid of hull vertices (even simpler: use average of all vertex positions projected into 2D)
        5. Point-in-polygon test: check if CoM projected point is inside the 2D footprint hull
        6. margin: minimum distance from CoM to any edge of footprint polygon (normalized by footprint area)
      - Use winding number or ray casting algorithm for point-in-polygon (stable against numerical edge cases per PITFALLS.md)
      - Use an epsilon of 1e-6 for "on edge" cases (treat as inside, as it's borderline stable)
    - `stable: bool` — true if CoM is inside footprint (including on edge)
    - `margin: f32` — 0.0 if unstable, otherwise ratio of min-edge-distance to sqrt(footprint_area) 
    - `contact_area: f32` — area of the 2D footprint polygon

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JS → WASM (compute_orientations input) | Unvalidated STL bytes (&amp;[u8]) and config JsValue arrive from browser JS |
| WASM → JS (Candidate[] output) | Processed results returned to browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Denial of Service | stl.rs STL parser | mitigate | Validate triangle count: reject STL with >5M triangles (prevents OOM). Handle malformed buffers gracefully — stl-io returns Result, unwrap with user-facing error. |
| T-01-02 | Tampering | scoring.rs | mitigate | f32 NaN propagation: ensure score_candidate filters NaN normals in mesh precompute (degenerate triangle filter). Add f32::is_finite() guard on final score. |
| T-01-02 | Denial of Service | hull.rs quickhull | mitigate | Guard against extreme vertex counts: if vertex count >5M, return error early. Quickhull is O(n log n) but degenerate inputs can cause pathological behavior. |
| T-01-02 | Elevation of Privilege | lib.rs config deserialization | mitigate | serde-wasm-bindgen from_value validates types at boundary. Reject unknown modes at runtime. Panic on invalid config fields — panic hook sends error to browser console. |
</threat_model>
