  <name>Task 1: Create Rust crate scaffolding with cdylib+rlib, wasm-bindgen, stl-io, serde deps</name>
  <files>
    core/Cargo.toml
    core/src/lib.rs
  </files>
  <action>
    Create `core/Cargo.toml` with:
    - package name `orient-core`, edition 2021
    - `[lib]` section with `crate-type = ["cdylib", "rlib"]` — cdylib for .wasm output, rlib for cargo test (per PITFALLS.md Pitfall 7)
    - dependencies:
      - `wasm-bindgen = "0.2"` — JS↔WASM bridge
      - `serde = { version = "1", features = ["derive"] }` — serialization derive macros
      - `serde-wasm-bindgen = "0.6"` — structured data serialization to plain JS objects (per ARCHITECTURE.md anti-pattern 2: NEVER use #[wasm_bindgen] struct handles)
      - `stl-io = "0.11"` — STL binary parsing, zero deps, verified compiles to wasm32-unknown-unknown (per spike 001)
      - `console_error_panic_hook = "0.1"` — dev dependency for Rust panic messages in browser console (per PITFALLS.md Pitfall 6)
    - `[profile.release]` with `opt-level = "s"` to minimize WASM binary size

    Create `core/src/lib.rs` with:
    - `use wasm_bindgen::prelude::*;`
    - `#[wasm_bindgen(start)]` function `init()` that calls `console_error_panic_hook::set_once();` — this runs automatically when WASM module loads
    - `#[wasm_bindgen]` function `compute_orientations(_bytes: &[u8], _config: &JsValue) -> JsValue` as a stub that returns an empty JS array via `serde_wasm_bindgen::to_value(&Vec::<()>::new()).unwrap()`
    - Import `serde_wasm_bindgen` for the to_value call so the build validates it compiles

    IMPORTANT design constraints from RESEARCH.md and orient-spec.md:
    - Single coarse-grained boundary: compute_orientations() takes &[u8] (raw STL bytes) + JsValue (config), returns JsValue (serialized candidates)
    - MUST use serde-wasm-bindgen for output, NOT #[wasm_bindgen] struct handles
    - Debug-assertion gate the panic hook: `#[cfg(debug_assertions)] console_error_panic_hook::set_once();` and `#[cfg(not(debug_assertions))] ()` — or just set_once() unconditionally (acceptable for dev builds, ~1KB binary cost)
  </action>
  <verify>
    <automated>cargo check 2>&1 && cargo check --target wasm32-unknown-unknown 2>&1</automated>
  </verify>
  <done>
    - Cargo.toml exists with cdylib+rlib crate types
    - lib.rs exists with #[wasm_bindgen(start)] init() and compute_orientations stub
    - `cargo check` passes (native target)
    - `cargo check --target wasm32-unknown-unknown` passes (WASM target)
  </done>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → web/node_modules | Third-party JS dependencies (vite, vite-plugin-wasm, typescript) fetched from npm |
| crates.io → core/target | Third-party Rust dependencies (wasm-bindgen, stl-io, serde) fetched from crates.io |
| developer machine → WASM binary | Build output is an executable binary consumed by the browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-SC | Tampering | npm packages (vite, vite-plugin-wasm, vite-plugin-top-level-await) | accept | Build toolchain dependencies are widely used, regularly audited. Risk of supply-chain attack is low; no sensitive data handled. |
| T-01-SC | Tampering | Cargo crates (wasm-bindgen, stl-io, serde, serde-wasm-bindgen) | accept | Established crates from official registries. stl-io has zero deps (per spike 001). serde/wasm-bindgen are foundational crates with extensive review. |
| T-01-01 | Information Disclosure | WASM binary (.wasm file served to browser) | accept | WASM binaries are inherently readable (though minified). orient-stl runs entirely client-side with no secrets. Mathematical algorithms are the IP — no mitigation applied (open source project). |
</threat_model>
